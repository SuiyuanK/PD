    need to be abc W or greater.

DESCRIPTION 
    Power assigned to given region/block in BLOCK_POWER_ASSIGNMENT construct is
    smaller than the sum of power assigned to sub-regions/sub-blocks under it. 

SEVERITY & IMPACT 
    In hierarchical power assignment, redhawk subtracts sum of sub-block/region

    block/region power is less than sum of sub-blk/sub-region power, redhawk will
    not assign any power to the block/region.

ACTIONS 
    Power specified for region/block should be greater than sum total power
    for its sub-block/region.

! CCL-064
    Instance is connected to multiple VDD domains.

DESCRIPTION
    This message reports the instances that are connected to multiple VDD
    domains. This is not handled in RedHawk 5.1 but will be supported in 5.2.

SEVERITY & IMPACT
    Warning. The biggest challenge in handling the instances that belong to 
    multiple VDD domains is how to correctly distribute the instance power
    to each domain. RedHawk 5.2 will get this info from Synopsys liberty files
    if available. Otherwise, RedHawk will do the following (as 5.1).

    (1) For static analysis: The instance power is evenly distributed to all 
        the domains specified in GSR (VDD_NETS). If only one domain is 
        specified in GSR, all the power will be assigned to that domain.
    
    (2) For dynamic analysis: The instance power is assigned to the primary 
        VDD domain specified in GSR. For example, given an instance which is 
        connected to VDD2 and VDD5, assuming VDD2 is ranked 2nd and VDD5 5th
        in VDD_NETS, the primary domain of the instance is VDD2 and all the
        instance power will be assigned to VDD2.

ACTIONS
    Users need to be aware of the implication of the results. Also, 
    if desired, users may manipulate VDD_NETS in GSR to control the power 
    assignments. In order to use the new multiple VDD suport in RedHawk 5.2, 
    users' .lib must have multi-rail info.

! CLK-451
    ClockRoot specified in GSR not found in the design. 

DESCRIPTION
    The design object specified in ClockRoot does not exist in the design.
    The possible causes include name case mismatch, incorrect/missing 
    hierarchical path, hierarchical divider or pin/bus delimiters mismatch,
    incomplete DEF files, or simply a typo.

SEVERITY & IMPACT
    Important to fix or this clock root will be ignored. The direct impact
    is to a wrong clock network extraction and frequency propagation.

ACTIONS
    Do the following checking:
    (1) any typo?
    (2) the specified design object does exist in DEF
    (3) the name case is consistent to DEF. Also, check if 
        NAME_CASE_SENSITIVITY is set to 0 in GSR. In this case, all design 
        object names are remapped to uppercase.
    (4) correct hierarchical path name with consistent hierarchical divider.
    (5) consistent pin delimiter and bus delimiter, if the clock root is 
        a pin or pinInst.

! CLK-812 
    Cell not defined in .lib

DESCRIPTION 
    You will get this warning when a particular cell present in the design is
    not defined in .lib files.  Redhawk uses timing arcs present in .lib for
    tracing paths. It tries to trace the clock tree from the clock root defined
    in gsr file. However if it comes across a cell which is not defined in .lib,
    it is not able to trace forward and gives this warning. Redhawk expects all
    cells in the clock path to be defined in the supplied .libs

SEVERITY & IMPACT 
    Warning. It blocks forward clock tracing from such cells.

ACTIONS 
    Supply the .lib with the particular cell defined in it.

! CLK-821
    STA file is not available for the design.

DESCRIPTION 
    The power for each net is calculated from the transition times 
    (“slew”) of each instance’s input/output pins, using the clock 
    frequency domain for all the instances that are specified in the STA
    output file.If this file is not available then the GSR keyword 
    CLOCK_ROOTS traces the nets from a specified clock root and finds 
    the respective clock domain for all the nets and instances.Specifies 
    the names of the clock roots(either net name or pin name) and the frequency.

SEVERITY & IMPACT 
    Warning generated during setup design. With neither STA and 
    CLOCK_ROOTS option given, the slew will be the default slew and 
    frequency will be the default GSR frequency. Also TW will be assumed 
    0 for all instances.Hence, the results will be inaccurate. More 
    important with respect to dynamic analysis.

ACTIONS 
    Include the keyword CLOCK_ROOTS in the GSR file:
    Syntax:
    CLOCK_ROOTS
    {
        <clock_root_name> <freq in Hz>
        ...
    } 
    Or else provide the STA file.

! DEF-108
   No SITE <unit> in LEF (or gsr FAO_ROW_SITE) for DEF/ROW creation 

DESCRIPTION 
   FAO_ROW_SITE specifies a particular set of “sites” (a placement 
   grid)by its name in DEF, to avoid conflicting and overlapping decap 
   insertions from multiple library row definitions.LEF “sites”, also 
   called “unit-tiles”, are defined in LEF and used in the DEF/ROW
   section to create “row boxes” to allow P/R tools to create multiple
   clusters using different cell grid libraries, and thus may create 
   overlapping ROW boxes. By default,without using FAO_ROW_SITE, FAO 
   takes all ROW grids defined in DEF to create placeable decap areas. 
   Optional; default: all defined DEF row grids.

   Example:
      For a DEF file ROW section containing several row definitions as follows:
      ROW CORE_ROW_37549 bonuscoreaxg 3547600 2680600 N DO 7793 BY 1 STEP 690 0;
      ROW CORE_ROW_52855 core 8927530 2680600 N DO 2054 BY 1 STEP 230 0 ;
      Using ‘FAO_ROW_SITE core’, only rows defined by the ‘core’ site 
      name are created for FAO decap placement.This warning is generated 
      when no SITE isspecified in the LEF (or gsr FAO_ROW_SITE) keyword
      is not used.

SEVERITY & IMPACT 
   Warning.FAO won't consider row! 

ACTIONS
   Define SITE in LEF.
   Syntax:
      Site
      SITE siteName
      CLASS {PAD | CORE} ;
      [SYMMETRY {X | Y | R90} ... ;] 
      SIZE width BY height ;
      END siteName
   And then define the following in GSR:
   Syntax:
      FAO_ROW_SITE {
         <LEF_site_name>
         ...
      }

! FLW-001
   Package parasitics are not annotated in the analysis

DESCRIPTION
   RedHawk expects package parasitic information annotated in the
   design before performing voltage drop analysis. Especially in dynamic analysis,
   It is very important to include  package parasitics to capture the L*di/dt impact
   on the overall voltage drop. Also, you will be able to accurately analyze
   the decap effectiveness in the design only if you include package parasitics
   in the analysis.

SEVERITY & IMPACT
   Important error, especially for dynamic analysis. Voltage drop results will
   be optimistic if package parasitics are not included

ACTIONS
   There are multiple methods available in RedHawk to annotate the package parasitics.
   Package spice netlist can be generated from package Extraction tools like SN-PSI
   or PakSi-E in the form of S-parameter or RLCK netlist. This package spice netlist
   can be included in GSR using PACKAGE_SPICE_SUBCKT keyword.

   If detailed package spice netlist is not available for analysis, user can also
   annotate lumped RLC values for package/bond-wire/pad-wire using
   "setup package/wirebond/pad" command.

! ITG-010    
   Pin delimiter ":" used in LEF macro <xyz> pin "abc:def".
   Change the pin delimiter to ";". 

DESCRIPTION 
   Redhawk found ":" character in pin names. As a result, it changed the default
   pin delimiter from ":" to ";". One application of pin delimter is to identify
   instance and pin name from clock root for clock tree tracing. It is only
   required when STA file is not available and user wants to use clock tree
   tracing for initial flow flush. In such cases, the clock roots with pin names
   will not be mapped to the design.

SEVERITY & IMPACT 
   Impacts analysis only when user is using clock tree tracing due to
   unavailability of STA file.

ACTIONS 
   If you are performing clock tree tracing, make sure to use ";" or some
   special character as pin delimiter in clock root and specify the same in
   GSR. Alternatively, avoid using ":" character in pin names.

! ITG-016
   Missing PG-arc definitions for some cells

DESCRIPTION
   For cells having multiple POWER and GROUND pins (mVmG cells), RedHawk require PG Arc
   information which defines the current flow between the POWER pin and GROUND pin.
   For example, if a cell has 2 POWER pins(VDD1 and VDD2) and 2 GROUND pins
   (VSS1 and VSS2), you should associate each power pin with the associated ground pin.
   (Example: VDD1->VSS1 & VDD2->VSS2). This information is passed to RedHawk as
   a custom lib file in LIB_FILES section.

SEVERITY & IMPACT
   Critical error. Voltage drop results will be impacted for mVmG cells.

ACTIONS
   Create a custom lib file containing PG ARC information like given below:
   cell vdds18hvt {
        pgarc {

                biasfet bus_vsss
                bus_vdd vss
                vdds bus_vsss
        }
   }

   The above file needs to be included in LIB_FILES GSR with a custom attribute.
   LIB_FILES  {
     <other lib files>
     custom.lib custom
   }

! NET-150
    Failed in building connectivity for net<name>.

DESCRIPTION
    Power or ground nets (specified in the power or ground net sections of the 
    GSR) that are present in the design but are not connected to the global 
    network that is driven by VDD or GND power supply sources are considered
    dangling wires. Their presence lead to a failure in building up a fully 
    connected power or ground network for the specified power or ground nets.
    This can happen due to lack of either physical or logical connectivity of
    these portions of the network to the rest of power or ground networks.

    The extraction checking finds that there exist disconnet from a top
    cell's P/G pin to all instance pins connected by the same net.

SEVERITY & IMPACT
    If the P/G net reported is used in the design, the simulation will fail.

ACTIONS
    This should be investigated to ensure data correctness. Inside RedHawk
    it is possible to debug this error message by using the "View->Highlight 
    unconnected wires/vias" feature following a static or dynamic analysis.
    This will highlight the portions of the power or ground networks that
    are dangling.  User may check the .ploc/.pad/.pcell file to make sure the
    pad locations for the reported net is specified and correctly placed.
    Search log file to see if any (CCL-056) or (CCL-065) messages exist.
    These messages points to pads which are detached and are potential
    source of this problem. Also, search for (NET-121) messages to check
    instance pins.

! PWR-100
    reactivated non-switching instances that were not covered in STA

DESCRIPTION
    RedHawk by default treats the instances that are not covered in STA file
    as non-switching instances (with zero frequency). But there are a few cases
    that such an instance will be reactivated as a switching instance. 

    The scenarios include:
    1) the instance is active in VCD file so it should switch,
    2) the instance has power specified BLOCK_POWER_FOR_SCALING in GSR,
    3) data mismatch in STA file. One example is the driver has zero frequency
       but its fanouts have running frequency. The incurred power will be
       annotated to the driver and force it to be a switching instance.

SEVERITY & IMPACT
    This is a data integrity issue that should be fixed. The reactivation 
    process potentially makes the static power more accurate but does not help
    the dynamic analysis because the timing windows are still not available.
    For these instances, the timing windows are set to the whole cycle.

ACTIONS
    Users may check out adsRpt/reactivate.inst for the list of instances
    that are reactivated. For these instances, the following steps
    can be taken to nail down the data issues:
    1) If VCD files were used to generate the toggle file, users may
       verify if these instances are switching in the VCD files. If yes,
       the problem is found and the inconsistency between VCD and STA files
       must be fixed.
    2) If BLOCK_POWER_FOR_SCALING is used, users may check if this instance
       or its master cell is specified with a fixed power. If yes,
       the problem is found and the inconsistency between 
       BLOCK_POWER_FOR_SCALING and STA files need to be fixed.
    3) Users may check the STA files to verify if any fanouts of these
       instances are switching but the driving pins of these instance are not.
       If yes, the problem is found and the data inconsistency in STA files
       must be fixed.
    4) If none of the above applies, please submit your data to Apache Design
       Solutions for further investigation.

! SPF-118
    The nets in <xx.spef> have incomplete drivers.

DESCRIPTION
    This message reports the number of nets which have inconsistent drivers 
    between SPF and DEF. Most likely this applies to a hierarchical net
    whose sibling nets are not fully covered by SPF files. Another often
    scenario is the confusing pin directions due to missing .lib, as a
    consequence of NET-412. The inconsistency is reported in
    adsRpt/apache.rcMismatch.
   
SEVERITY & IMPACT
    In the case of only partial parasitics is available, the partial
    parasitics (from SPF files) will be ignored, and RedHawk will estimate 
    the parasitics using Steiner trees instead.

    In the case of confusing pin directions, the SPF file is honored.

    The severity of this warning depends on the percentage of nets that are
    having problems. Also, in general, Steiner tree estimation on a 
    hierarchical net is less accurate than the estimation on a local net.
    So there is some accuracy penalty when the SPF files are not complete.

ACTIONS
    Users need to check out the nets listed in adsRpt/apache.rcMismatch and 
    investigate what cause the mismatch between SPF and DEF files. Most likely
    the SPF files are not complete or .lib is not complete, but it could also 
    because the design objects in SPF cannot be mapped to DEF properly due to 
    inconsistent name case sensitivity, mismatched hierarchical divider, or 
    inconsistent pin/bus delimiter.

    (5) consistent pin delimiter and bus delimiter, if the clock root is 
        a pin or pinInst.

! LIB-142 
   Pratio file error

DESCRIPTION
    ERROR(LIB-142) is the error message identified during reading Pratio file.
    They are a set of messages sharing the same message ID: LIB-142, inlcuding:
      1). "number of fields <num> is not 2 or 3"    
          Each line of Pratio file is supposed to have 2 or 3 fields.
      2). "Cell <cell_name> doesn't exist"
          The cell_name specified after "#cell" does not exist in RedHawk db.
      3). "no cell specified"
          No cell line "#cell   cell_name" in Pratio file.    
      4). "pin <pin_name> doesn't exist"
          Cell doesn't contain a pin named "pin_name"
      5). "first field <> is not an integer"
           First field of each line(except the #cell and #pin line) must be an 
           integer.
      6). "second field <> is not an integer"
           Second field of each line(except the #cell and #pin line) must be an
           integer.
      7). "second field <> is negative"
           Second field of each line(except the #cell and #pin line) must be a 
           positive integer.
      8). "no pin specified"
           No pin line "#pin   pin_name" in Pratio file.
      9). "invalid pin index <num> (valid index range [0, num])"
           Pin index(first field of each line) is not valid.

SEVERITY & IMPACT
    RedHawk fails to read the Pratio file.

ACTIONS
    Fix the data errors in Pratio file according to the messages.

    DYNAMIC_SIMULATION_TIME.  So if END_TIME is set, RedHawk modifies the END_TIME 
    as (START_TIME + DYNAMIC_SIMULATION_TIME).

SEVERITY & IMPACT
    Warning. User will think that power calculation results are for the specified 
    START and END time in the GSR file. But actually the results are for 
    START time and (START + DYNAMIC_SIMULATION_TIME)

ACTIONS
    User do not have any control on this.
